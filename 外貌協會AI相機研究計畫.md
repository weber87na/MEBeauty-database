## 一、研究問題與目的

### 1.1 研究背景

現在大家很常在 Instagram、TikTok、Threads、LINE 或其他社群平台上分享自拍照、聚會照和短影音。很多人拍照時不只是想「留下紀錄」，而是希望照片看起來好看、有趣、能被朋友轉傳，甚至能成為聚會中的話題。也因為這樣，美顏相機、濾鏡 App 和自拍特效已經變成很常見的工具。

不過，目前大多數美顏相機的功能其實差不多，主要都是磨皮、瘦臉、大眼、調色、加濾鏡。這些功能雖然可以讓照片更好看，但使用者通常只是自己選效果，相機本身不太會根據畫面內容做出有趣的反應。也就是說，現在的美顏相機多半是「使用者控制相機」，而不是「相機主動吐槽使用者」。

本研究想開發的「外貌協會 AI 相機」就是要突破這個限制。這個 App 會根據拍照時的人臉分數、表情狀態和畫面情境，自動決定要輸出哪一種照片結果。它不是單純把人拍漂亮，而是把「外貌協會」這個概念做成一個會判斷、會嫌棄、會害怕、會求生的趣味相機。

本 App 主要會有三種輸出效果：

1. **正常美照**：如果系統判斷使用者顏值分數高，就輸出正常美化後的照片，並加上高顏值認證浮水印。
2. **殘酷無頭照**：如果系統判斷使用者分數普通，就故意把頭部去除，並用影像修補技術把背景補起來，做出「好不好看決定是否有頭」的惡搞效果。
3. **畏懼美顏**：如果系統偵測到使用者表情很兇、很生氣或很有威脅感，相機會「屈服求生」，無視原本顏值分數，強制套用最大美顏，並在畫面上呈現眼球發抖的害怕動畫。

這個計畫不只是做一個搞笑 App，也希望透過這種反諷方式，讓使用者思考現在社群媒體中常見的「顏值焦慮」和「外貌評價」問題。也就是說，本研究的核心不是要證明誰好看或不好看，而是把顏值評分這件事變成一種可互動、可分享、也有一點諷刺意味的娛樂體驗。

### 1.2 研究目的

本研究目標是開發一套可以實際操作的行動相機 App 原型，讓系統能從相機畫面中分析人臉，並自動產生對應的照片效果。具體目的如下：

1. **建立人臉分數判斷功能**：參考 BeautyPredict 專案與 SCUT-FBP5500 人臉顏值資料庫，讓系統可以輸出一個 0 到 5 分之間的人臉分數 [1][2]。
2. **加入表情與威脅判斷**：除了顏值分數外，也判斷使用者是否露出生氣、兇狠或威脅性表情，避免系統只靠顏值做決定。
3. **設計趣味互動效果**：讓 App 可以依照判斷結果輸出正常美照、殘酷無頭照或畏懼美顏，並搭配浮水印、排行榜和動態 UI。
4. **規劃前端與雲端影像生成架構**：前端初步使用原生 HTML、CSS、JavaScript 實作，負責拍照、預覽和互動介面；雲端伺服器負責較吃效能的人臉分數推論、頭部遮罩和呼叫 `gpt-image-2` 進行影像生成或修補。
5. **建立測試與評估方式**：測試 App 是否能穩定拍照、正確呼叫雲端服務、順利產生照片，並記錄延遲、誤判率和使用者體驗。

## 二、文獻探討

### 2.1 人臉分數與吸引力預測技術

人臉吸引力預測（Facial Beauty Prediction）簡單來說，就是讓電腦看一張人臉照片後，預測這張臉在人類主觀評分中大概會得到幾分。早期的做法比較依賴人工設計特徵，例如臉部對稱性、五官距離、臉型比例等，再用統計方法或機器學習模型進行預測。

近年來，卷積神經網路（CNN）常被用在人臉影像分析上。CNN 的優點是它不需要完全靠人工指定特徵，而是可以從大量圖片中自己學習哪些臉部特徵和分數有關。BeautyPredict 這個 GitHub 專案就是一個可參考的例子。它使用 SCUT-FBP5500 資料庫進行人臉分數預測，並測試 AlexNet、ResNet-18、ResNeXt-50 等模型，也包含 Label Distribution Learning 的做法 [1][2][3]。

SCUT-FBP5500 資料庫共有 5,500 張正面人臉照片，分數範圍是 1 到 5 分，並包含不同性別、族群和年齡的人臉。BeautyPredict 專案說明中提到，使用較深的 CNN 模型可以得到不錯的預測效果，其中以 ResNet 微調搭配 Label Distribution Learning 的模型，在該資料集上的 Pearson correlation coefficient 約可達 0.95 [1][2][3]。

本研究會參考 BeautyPredict 的方式，把人臉分數模型當作 App 的核心判斷依據之一。不過，本研究不會把這個分數當成真正客觀的美醜標準，而是把它當成娛樂互動用的參數。也就是說，這個分數是用來控制 App 要不要吐槽、要不要修圖、要不要觸發無頭照效果，而不是用來評價人的真正價值 [1]。

### 2.2 人物分割與頭部遮罩技術

要做出「殘酷無頭照」，系統需要先知道照片中的頭在哪裡，然後把頭部區域標記成一個遮罩（Mask）。遮罩可以想像成一張黑白圖，白色區域代表要被移除的地方，黑色區域代表保留的地方。

這個部分可以用人物分割（Semantic Segmentation）或人臉定位技術來完成。人物分割可以把人和背景分開，人臉偵測可以找出臉部位置，而臉部 landmarks 可以更細緻地標出眼睛、鼻子、嘴巴、下巴等位置。若要把整個頭部移除，還需要把臉框往上、往左、往右和往下適度擴張，避免只移除臉而留下頭髮或下巴。

在 App 初期實作時，可以先採用較簡單的方式：用人臉偵測找出臉部框，再根據框的位置推估頭部範圍。等基本功能穩定後，再加入 MediaPipe Selfie Segmentation、Face Mesh 或 Segment Anything Model（SAM）這類分割工具，讓遮罩邊界更準確，減少殘留臉部輪廓的問題 [4][5]。

### 2.3 影像修補與背景重建

當系統把頭部區域移除後，照片中會留下一塊空白區域。這時候就需要使用影像修補（Image Inpainting）技術，把空白區域補成合理的背景。舉例來說，如果使用者站在教室前面拍照，系統把頭移掉後，就要把原本頭後面的白板、牆壁或背景補回來，讓照片看起來像一開始就沒有頭。

傳統影像修補方法多半是從周圍像素複製紋理，適合修掉小瑕疵，例如刮痕或小物件。但如果要移除整個頭部，空缺區域比較大，傳統方法容易出現模糊、破圖或背景不連續。

近年的深度學習影像修補技術效果較好，本研究規劃使用 OpenAI 的 GPT Image 系列模型，目標以 `gpt-image-2` 作為雲端影像修補與照片生成工具。若實作當下雲端算力吃緊，則先改用官方可用的 GPT Image 模型作為替代，例如 `gpt-image-1.5` 或 `gpt-image-1` [6][7][8]。

使用 GPT Image 系列模型的好處是開發速度較快，不需要一開始就自行訓練大型影像修補模型。系統可以先把原始照片、頭部遮罩和文字提示送到後端，再由後端呼叫影像生成 API，要求模型「移除頭部並自然補齊背景」或「套用誇張美顏並保留原本人物特徵」。這樣比較符合本專案用 Vibe Coding 快速做出可展示 App 的目標 [7][8]。

## 三、研究方法與系統設計

### 3.1 系統架構與實作流程

本研究會把 App 設計成一個「拍照 -> 分析 -> 判斷 -> 呼叫影像生成 -> 顯示結果」的流程。使用者按下拍照後，前端會把影像送到後端服務，後端會做人臉分數推論、表情威脅判斷、頭部遮罩，並視情況呼叫 `gpt-image-2` 產生修補或美顏結果，最後把完成的照片回傳給前端顯示 [1][2][7][8]。

整體流程如下：

1. 使用者開啟 App 並啟動相機。
2. 前端取得相機畫面，使用者按下拍照。
3. App 將照片壓縮後送到後端 API。
4. 後端偵測人臉並計算人臉分數。
5. 後端判斷是否有憤怒、凶狠或威脅性表情。
6. 決策函式根據分數決定輸出狀態。
7. 如果需要無頭照，系統產生頭部遮罩，並呼叫 `gpt-image-2` 或可用的 GPT Image 模型進行 Image Inpainting [7][8]。
8. 前端顯示最後照片、浮水印、動畫和分享按鈕。

系統模組設計如下表：

| 系統模組 | 功能描述 | 本研究系統實作方式 |
| --- | --- | --- |
| **影像擷取模組** | 從手機鏡頭取得即時畫面與拍照影像。 | 使用原生 JavaScript 的 `getUserMedia` 或檔案上傳方式擷取畫面，拍照後壓縮成 JPEG 或 WebP，再送到後端。 |
| **人臉分析模組** | 找出照片中的人臉，並計算人臉分數。 | 參考 BeautyPredict 的 CNN 模型流程，輸出 $S_{beauty} \in [0,5]$ 的分數 [1][2]。 |
| **表情判斷模組** | 判斷使用者是否有生氣、兇狠或威脅性表情。 | 使用表情分類模型或情緒分數，輸出 $E_{threat} \in [0,1]$ 的威脅指數。 |
| **決策判定模組** | 根據人臉分數與威脅指數決定照片效果。 | 使用規則式分數累積法，分數達門檻後觸發正常美照、殘酷無頭照或畏懼美顏。 |
| **影像後製模組** | 產生無頭照、美顏照與浮水印。 | 後端產生頭部 Mask，並呼叫 `gpt-image-2` 進行背景修補或誇張美顏；前端負責浮水印、結果頁和分享按鈕 [7][8]。 |
| **紀錄模組** | 記錄每次判斷結果與觸發原因。 | 儲存時間、分數、觸發狀態、延遲和錯誤訊息，方便後續調整模型。 |

### 3.2 程式與模型設計

本 App 預計分成前端、後端和模型服務三個部分。初期開發重點會放在「先做出可以跑的 Web 版原型」，因此前端先不使用大型框架，而是以原生 HTML、CSS、JavaScript 實作，降低環境設定難度，也方便在課堂展示時直接用瀏覽器測試。

前端主要負責相機畫面、拍照按鈕、結果頁、浮水印、排行榜、分享功能和眼球發抖動畫。實作上會使用原生 JavaScript 呼叫瀏覽器相機 API，將畫面繪製到 `<canvas>`，再把照片轉成 Blob 或 Base64 傳給後端。因為使用者會直接操作前端，所以畫面需要簡單、直覺，而且要有明顯的娛樂效果。若後續要正式做成手機 App，再考慮改成 React Native、Flutter 或原生 iOS、Android 版本。

後端可以使用 Python FastAPI 或 Node.js Express 建立 API。後端收到照片後，會呼叫不同模型服務，包含人臉偵測、人臉分數預測、表情判斷、頭部遮罩和影像生成。人臉分數模型可部署在伺服器上；影像修補與誇張美顏則優先透過 OpenAI 影像生成 API 呼叫 `gpt-image-2` [7][8]。

人臉分數模型會參考 BeautyPredict 的架構。初期可以先使用預訓練 CNN 模型，例如 ResNet-18，搭配 SCUT-FBP5500 資料集微調。輸入是一張裁切後的人臉照片，輸出是 0 到 5 分之間的分數。未來如果要提升準確度，可以再改用 ResNeXt-50 或 Label Distribution Learning [1][2][3]。

`gpt-image-2` 的呼叫會由後端處理，不直接放在前端，避免 API key 暴露。前端只會送出照片與模式需求，例如 `headless_photo` 或 `survival_beauty`；後端再組合提示詞、原始影像與遮罩，送到影像生成模型。回傳結果可先用 Base64 或暫存圖片 URL 傳回前端顯示 [7][8]。

系統主要可調整參數如下：

1. `conf`：模型信心門檻，太低代表系統不確定，暫時不觸發效果。
2. `cooldown`：冷卻時間，避免使用者連續拍照時一直重複觸發同一效果。
3. `Th_high`：高顏值門檻，例如 3.8 分以上進入正常美照。
4. `Th_low`：普通顏值下限，例如 2.0 分以上但未達高顏值時進入嫌棄判斷。
5. `Th_threat`：威脅門檻，例如威脅指數 0.65 以上代表使用者表情太兇。
6. `score_threshold`：總分門檻，累積分數達 3 分以上就觸發正式效果。
7. `image_model`：影像生成模型名稱，預設規劃為 `gpt-image-2`，實作時依 OpenAI API 可用模型調整 [7][8]。
8. `image_quality`：影像生成品質設定，初期可用低成本快速模式，正式輸出再使用較高品質。

### 3.3 決策邏輯與行為判定機制

本研究會設計一個名為 `estimate_behavior` 的判定函式。這個函式會接收兩個主要輸入：人臉分數 $S_{beauty}$ 和威脅指數 $E_{threat}$。系統不會只看單一條件，而是把不同條件加總後再決定結果。

判定條件如下：

1. 若 $S_{beauty} \ge Th_{high}$，代表系統判斷為高顏值，加 2 分，輸出正常美照與高顏值認證浮水印。
2. 若 $Th_{low} \le S_{beauty} < Th_{high}$，代表系統判斷為普通，加 2 分，進入嫌棄狀態，呼叫分割與 Inpainting 產生殘酷無頭照。
3. 若 $E_{threat} \ge Th_{threat}$，代表系統偵測到憤怒、兇狠或不友善表情，加 1 分。
4. 若威脅指數加上環境干擾後達標，例如畫面很暗、臉太靠近鏡頭、表情太兇，系統再加 1 分，直接進入畏懼美顏模式。

簡化後的判定概念如下：

```python
def estimate_behavior(S_beauty, E_threat, W_env, now, last_trigger_at, config):
    if now - last_trigger_at < config.cooldown:
        return "cooldown"

    score = 0
    state = "preview"
    reasons = []

    if S_beauty >= config.Th_high:
        score += 2
        state = "normal_beauty"
        reasons.append("高顏值")
    elif config.Th_low <= S_beauty < config.Th_high:
        score += 2
        state = "headless_photo"
        reasons.append("普通顏值，觸發嫌棄效果")

    if E_threat >= config.Th_threat:
        score += 1
        reasons.append("表情具有威脅感")

    survival_score = 0.7 * E_threat + 0.3 * W_env
    if survival_score >= config.Th_survival:
        score += 1
        state = "survival_beauty"
        reasons.append("相機判定需要求生")

    if score >= config.score_threshold:
        return state, score, reasons

    return "preview", score, reasons
```

這樣設計的好處是判斷過程比較清楚。系統不是直接說「這張照片要無頭」，而是可以記錄原因，例如「顏值分數落在普通區間」、「表情威脅感偏高」、「目前冷卻時間尚未結束」。這些紀錄可以幫助後續調整門檻，也能讓報告中的實驗結果更容易說明。

### 3.4 實驗設計與驗證方法

本研究會用三個層次測試系統：功能測試、情境測試和效能測試。

**一、功能測試**

功能測試主要確認 App 的每個基本功能是否能正常運作，包括：

1. 相機是否能正常開啟與拍照。
2. 照片是否能成功送到雲端 API。
3. 後端是否能回傳人臉分數與威脅指數。
4. 系統是否能正確產生頭部遮罩。
5. 呼叫 `gpt-image-2` 後，Inpainting 是否能補出合理背景 [7][8]。
6. 前端是否能顯示正常美照、殘酷無頭照與畏懼美顏。
7. 浮水印、排行榜與分享功能是否能正常使用。

**二、情境測試**

情境測試會設計不同拍照情境，觀察系統是否能做出合理反應。例如：

1. 使用者正常微笑拍照，檢查是否輸出正常美照。
2. 使用者表情普通，檢查是否可能觸發無頭照。
3. 使用者故意露出生氣或威脅表情，檢查是否觸發畏懼美顏。
4. 多人同時入鏡，檢查系統會選擇最大臉、中央臉，或提示使用者重新拍攝。
5. 光線不足、臉部遮擋、背景複雜時，檢查模型是否容易誤判。

**三、效能測試**

效能測試主要確認 App 是否能達到實際使用需求。測試項目包含：

1. 從按下拍照到收到判斷結果的時間，目標盡量控制在 500 毫秒左右。
2. 呼叫 `gpt-image-2` 產生高品質無頭照或畏懼美顏的時間，目標控制在 2 到 5 秒內 [7][8]。
3. 人臉分數模型的平均誤差，例如 MAE。
4. 影像修補品質，例如 PSNR、SSIM 或人工主觀評分。
5. 誤判率與漏判率，例如該觸發畏懼美顏卻沒有觸發，或不該觸發卻觸發。
6. 手機前端是否會卡頓、動畫是否順暢。

## 四、預計達成的成果與應用價值

### 4.1 系統實作成果

本研究最後預計完成一個可操作的 App 原型。使用者可以用手機打開相機、拍照，系統會自動分析照片並產生結果。這個結果可能是正常美照、殘酷無頭照，也可能是畏懼美顏。

預計完成的功能包含：

1. 手機相機拍照與預覽。
2. 人臉偵測與人臉分數預測。
3. 表情威脅判斷。
4. 頭部遮罩產生。
5. Image Inpainting 背景修補。
6. 正常美照、殘酷無頭照、畏懼美顏三種輸出模式。
7. 搞笑浮水印與結果頁。
8. 社群分享與排行榜雛形。
9. 判斷紀錄與系統日誌。

### 4.2 技術成果與可量化指標

本研究預計產生以下技術成果：

1. 一套使用原生 HTML、CSS、JavaScript 製作的可展示 Web App 原型，能完成從拍照到結果輸出的完整流程。
2. 一個參考 BeautyPredict 設計的人臉分數模型 [1][2]。
3. 一套用於頭部移除的遮罩產生流程。
4. 一個呼叫 `gpt-image-2` 或可用 GPT Image 模型產生無頭照與畏懼美顏的影像生成流程 [7][8]。
5. 一個以 `estimate_behavior` 為核心的決策函式。
6. 一組可調整的系統參數，例如 `conf`、`cooldown`、`Th_high`、`Th_low` 和 `Th_threat`。

預計量化指標如下：

1. 人臉分數模型 MAE 目標控制在 0.35 到 0.50 分之間。
2. 快速判定延遲目標小於或接近 500 毫秒。
3. 高品質無頭照或畏懼美顏生成時間目標為 2 到 5 秒。
4. 頭部遮罩主要區域覆蓋率目標達 90% 以上。
5. Inpainting 結果需通過人工檢查，不應有明顯破圖或背景斷裂。
6. 情境測試後能整理出誤判率、漏判率與可改善項目。

### 4.3 應用價值與後續發展

這個 App 的主要應用價值在娛樂和社群互動。它可以用在朋友聚會、校園活動、派對破冰、品牌活動或短影音創作。因為輸出結果具有吐槽感和反差感，使用者比較容易想把結果分享給朋友，也比較容易形成話題。

後續可以發展的方向包含：

1. 加入多張連拍或短影片分析，讓判斷不只靠單張照片。
2. 增加更多相機人格，例如毒舌模式、膽小模式、偶像模式。
3. 加入使用者回饋，讓系統可以調整不同人的喜好。
4. 串接社群平台 API，讓使用者可以一鍵分享結果。
5. 建立活動排行榜，例如今日最高顏值、最兇表情、最慘無頭照。
6. 未來若要正式上架，需要補充隱私權政策，清楚說明照片是否儲存、如何刪除、是否用於模型改善。

## 五、協作開發技術說明：Vibe Coding 的實踐

本專案在開發過程中會使用 Vibe Coding 的方式協助完成原型。這裡所說的 Vibe Coding，是指開發者先用自然語言描述想要的功能、流程和規則，再讓生成式工具協助產生程式雛形，接著由開發者修改、測試和整合。本研究會特別把 Vibe Coding 用在原生 JavaScript 前端、後端 API、`estimate_behavior` 判定函式，以及 `gpt-image-2` 影像生成呼叫流程上。

本研究可以先從一個簡單的 Web 影像通訊架構開始。例如前端先用原生 JavaScript 開啟相機，把畫面畫到 `<canvas>`，按下拍照後轉成 Blob 或 Base64，再用 `fetch()` 傳送到後端。後端收到圖片後，再進行人臉分數判斷、表情判斷和影像生成。這個階段的重點不是馬上完成整個 App，而是先建立「拍照、送出、分析、呼叫模型、回傳結果」的基本流程。

接著，透過 Vibe Coding 把這個基本架構逐步改成外貌協會 AI 相機需要的功能。開發者可以用自然語言描述規則，例如：「如果顏值分數高就輸出正常美照」、「如果分數普通就移除頭部並修補背景」、「如果偵測到憤怒表情就強制套用最大美顏」。影像生成部分則描述為：「後端收到 `headless_photo` 狀態時，使用原圖與頭部遮罩呼叫 `gpt-image-2`，要求模型自然補齊背景；收到 `survival_beauty` 狀態時，呼叫 `gpt-image-2` 產生誇張美顏版本」。生成出的程式再由開發者檢查邏輯、調整參數並接上實際 API。

本專案中 Vibe Coding 主要會協助以下工作：

1. 快速建立原生 JavaScript 前端相機畫面與結果頁。
2. 產生後端 API 的初版程式。
3. 協助撰寫 `estimate_behavior` 判定函式。
4. 整合 BeautyPredict 風格的人臉分數模型 [1][2]。
5. 串接 `gpt-image-2` 或官方可用 GPT Image 模型的 Image Inpainting 流程 [7][8]。
6. 協助產生測試案例與錯誤處理程式。

整體而言，Vibe Coding 可以加快原型開發速度，但最後仍需要開發者實際測試、修正和驗證。特別是本 App 牽涉到人臉照片、主觀分數、OpenAI API 呼叫和社群分享，因此除了功能能不能跑之外，也要注意 API key 保護、照片上傳告知、隱私、誤判、使用者感受和娛樂尺度 [7][8]。
